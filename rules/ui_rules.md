Весь UI проекта написан на фреймворке NiceGUI

## Общие принципы

- Только базовые компоненты, с минимумом `.classes` и без `.styles`. Весь дизайн будет только на последнем этапе
- UI должен быть настолько простым, насколько это возможно для текущей задачи.
- Не пиши интерфейс «на вырост»: не добавляй вкладки, фильтры, режимы, панели и заготовки под будущий функционал, если
  они не нужны прямо сейчас.
- Перед добавлением нового элемента в UI задавай себе вопрос: "Он действительно нужен для текущего сценария?"

## Структура UI

- Избегай глубокой вложенности `row`, `column`, `card`, `grid` и других контейнеров.
- Если layout становится трудно читать — разбей его на маленькие функции с говорящими именами.
- Каждый крупный логический блок интерфейса выноси в отдельную функцию.
- Не держи весь экран в одной огромной функции.
- Не дублируй фрагменты UI; повторяющийся блок выноси в переиспользуемую функцию.
- Не смешивай разметку, бизнес-логику и доступ к данным в одном месте.

## Состояние

- Состояние должно быть локальным и явным.
- Не храни состояние в случайных переменных модуля.
- Не используй глобальные переменные для управления UI.
- Если структура состояния известна заранее — описывай её явно, а не через произвольные `dict`.
- UI должен отображать состояние, а не вычислять бизнес-логику по месту.

## Обработчики событий

- Обработчики событий должны быть короткими и однозначными.
- Не размещай бизнес-логику прямо внутри `on_click`, `on_change` и других UI-событий.
- Одно событие — одна понятная ответственность.

## Ошибки и обратная связь

- UI не должен молча скрывать ошибки.
- Пользователь должен получать понятное сообщение об ошибке без технического шума.
- Детали ошибки должны логироваться, а не показываться пользователю напрямую.
- `ui.notify(...)` используй только для короткой и простой обратной связи.
- Не используй `ui.notify(...)` как замену нормальной обработке ошибок.
- Сообщения в интерфейсе должны быть консистентны по языку и стилю.

## Async и долгие операции

- Не блокируй интерфейс синхронными операциями.
- Любая работа с сетью, файлами, БД и внешними сервисами не должна подвешивать UI.
- Async-обработчики должны быть действительно асинхронными.
- IO sync код в UI-обработчиках допускается только через `run_in_executor` / `to_thread`.
- Если операция может выполняться заметное время — пользователь должен видеть, что она началась и когда завершилась.

## Именование

- Имена функций, строящих UI, должны отражать назначение блока, а не тип контейнера.
- Предпочитай имена вроде `render_user_form`, `build_orders_table`, `create_filters_panel`, а не `make_column`,
  `create_row`, `build_card`.
- Переменные компонентов должны называться по смыслу: `name_input`, `save_button`, `status_label`.
- Избегай имён `x`, `item`, `obj`, `tmp`, если это не локальный тривиальный случай.
