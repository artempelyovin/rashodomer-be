## Комментарии

- Закомментированный код недопустим - для этого есть git history
- Минимум комментов в коде! Код должен быть максимально самодокументриуемым
- Комментарии допустимы только там, где без них логика действительно неочевидна
- Перед тем как писать комментарий, задавай себе вопрос: "Оставил бы такой комментарий senior Python-разработчик в этом месте?"
- `docstring` допускается только для публичных API и только если логика нетривиальная, объёмная или требует контракта использования.

## Типы

- Все функции, методы и публичные атрибуты должны быть типизированы.
- Используй современный синтаксис Python 3.10+: `str | None`, `list[str]`, `dict[str, int]` и т. п.
- Избегай неявных `Any`; используй его только тогда, когда тип действительно невозможно уточнить
- Для функций, которые могут ничего не возвращать, используй `-> None` явно
- Вместо `typing.Collections`, `typing.Mapping` используй `collections.abc.Collections` и `collections.abc.Mapping`
- При описании enums наследуйся от `StrEnum`, а не от `(str, Enum)`

## Сложность кода

- **KISS** — решение должно быть настолько простым, насколько это возможно для текущей задачи.
- Не пиши код «на вырост»: не добавляй абстракции, флаги, параметры, хуки или расширяемость, которые не нужны прямо
  сейчас (YAGNI).
- Если функция получается слишком длинной или ветвистой — разбей её на маленькие, однозначные части с говорящими
  именами.
- Не создавай классы там, где достаточно функции или простого модуля.
- Избегай магических методов (кроме стандартных `__init__`, `__repr__`) без веской причины.
- Не используй метапрограммирование, декораторы с параметрами, динамическое создание типов и прочие сложные техники,
  если этого не требует библиотечное API.

## Логгирование

- Используй стандартный `logging`
- Каждый файл имеет свой логгер (`logger = logging.getLogger(__name__)`)
- Явно выбирай уровень:
	- `debug`—внутренняя трассировка
	- `info`—бизнес-события
	- `warning`—нештатная, но обработанная ситуация
	- `error`/`exception`—только реальные ошибки
- Для логов никогда не используй `print()`

## Ошибки

- Предпочитай кастомные исключения.
- Имена кастомных исключений должны оканчиваться на `Error`, а не `Exception`.
- При перехвате и повторном возбуждении исключения сохраняй контекст: `raise CustomError(…) from original_exc`.
- Не глуши исключения молча; как минимум логируй их через `logger.exception(…)`.
- Не используй слишком широкие перехваты вроде `except Exception`, если можно поймать конкретный тип.
- Сообщение об ошибке должно быть встроено в сам класс исключения.
- В конструктор исключения передавай только данные для подстановки, а не готовую строку целиком, например: `raise UserNotFoundError(user_id=123)`.

## Форматирование строк

- Используй `fstring`
- Внутри `f-string` подставляй только простые выражения
- Если выражение становится сложным, вычисли его заранее в отдельной переменной

## Импорты

- Не используй wildcard-импорты: `from module import *`
- Импортируй модуль, а не объект, если имя конфликтует или важен контекст
- Используй только абсолютные импорты

## Именование

- Константы—`UPPER_SNAKE_CASE` на уровне модуля
- Приватные атрибуты—один underscore: `_name`
- Двойной underscore используй только для специальных случаев магических методов
- Булевы переменные и функции называй с префиксами `is_`, `has_`, `can_`
- Избегай однобуквенных имён вне list comprehension и математических формул
- Избегай имён, которые затеняют встроенные функции (`list`, `dict`, `str`)

## Структуры данных

- Для бизнес-логики предпочитай `dataclass`
- Для API-слоя и валидации предпочитай `pydantic`
- Не используй сырой `dict` там, где структура данных известна заранее и может быть описана явно

## Async

- IO sync код в async коде только через `run_in_executor`/`to_thread`!
- Не блокируй event loop синхронными операциями!

## Линтеры

- Проект проверяется `ruff` и `ty`; в идеале, код должен соответствовать их правилам сразу, а не после ручной правки
- `# noqa` и `# type: ignore`—крайняя мера
- Любое подавление проверки должно быть точечным и обоснованным

## RED флаги

- Использование глобальных переменных (явный `global` в функции)!
